# Cast-to-NFT 説明書レビュー

## 総評

元の説明書は、技術要素が一通り並んでいて実装の方向性は見えますが、実際に開発を始めるための仕様書としてはまだ不足があります。特に「誰のためのアプリか」「どこまでを MVP とするか」「失敗ケースをどう扱うか」「なぜその設計にするか」が弱く、実装時に判断がぶれやすい状態です。

## 良い点

- プロダクトのコンセプトが一読で伝わる
- 採用技術が具体的で、初期実装のイメージを持ちやすい
- Farcaster、Neynar、Pinata、Base など主要な外部依存が整理されている
- 画面遷移と API ルートのたたき台があり、全体像を掴みやすい

## 主な改善ポイント

### 1. プロダクト要件と実装要件が混ざっている

現状は「何を実現したいか」と「どう作るか」が同じ粒度で並んでいます。これだと、実装中に仕様変更が入ったとき影響範囲を判断しにくくなります。

必要なのは、少なくとも以下の分離です。

- プロダクト概要
- 対象ユーザーと提供価値
- MVP 範囲
- 機能要件
- 非機能要件
- 技術設計

### 2. 成功条件が曖昧

「Cast を NFT にできる」だけでは、どこまでできたら完成なのか判断しづらいです。

たとえば MVP の完了条件として、以下のような明文化が必要です。

- Cast URL または MiniApp コンテキストから Cast を取得できる
- 作者本人のみミント開始できる
- 画像と metadata が IPFS に保存される
- Base Sepolia で NFT を発行できる
- ミント完了後に共有導線が表示される

### 3. 認可・権限まわりの説明が不足している

作者確認ロジックは触れられていますが、実装上の判断がまだ粗いです。特に以下は明記したほうが安全です。

- Neynar のアドレス紐付けが複数 FID を返した場合の扱い
- Farcaster 側のウォレット更新遅延がある場合の UX
- 作者以外は `createAndMint` できないことをコントラクトでも担保するのか、アプリ層だけなのか
- URL ベースの一意判定ではなく、Cast hash を主キーにするかどうか

### 4. コントラクト仕様に曖昧さがある

実装時に事故になりやすいポイントがあります。

- `castUrl` を一意キーにすると URL 表記揺れの影響を受けやすい
- `transfer` は送金失敗やガス制約で問題になりやすく、pull payment の検討余地がある
- ERC-1155 である理由と、ERC-721 ではなく ERC-1155 を選ぶ理由が明文化されていない
- 初回ミントと追加ミントの在庫管理方針が曖昧
- 永続的な supply cap を設けるかどうかが不明

### 5. 失敗ケースと例外系が弱い

実装では正常系よりも異常系の定義が重要です。最低限、以下を仕様化すると開発が安定します。

- Cast が取得できない
- Cast に画像がない
- 投稿が削除済み
- 作者確認に失敗
- IPFS アップロード失敗
- トランザクション失敗
- Base Mainnet ではなく別チェーン接続中

### 6. UI フローが「操作説明」で止まっている

画面遷移はありますが、各画面で何を判断させるか、何が不安になるか、何を見せるべきかが不足しています。

特に以下を補うと、使いやすさがかなり上がります。

- ミント前の確認項目
- 手数料や価格の表示方法
- 失敗時の再試行導線
- ミント後の次アクション

### 7. 運用面の記述が不足している

プロトタイプなら動きますが、公開アプリとしては以下も必要です。

- API rate limit 対策
- 環境変数の責務分離
- ログと監視
- abuse 対策
- メタデータ再生成を許すかどうか

## 改善方針

改善版では次の方針で再構成するのが適切です。

- 最初に「誰の、どんな課題を解くか」を明記する
- MVP 範囲と後続フェーズを切り分ける
- 機能要件と技術設計を分離する
- 認可、例外処理、制約を先に仕様化する
- 実装担当者が迷わない粒度で API、画面、データモデルを定義する

## 結論

元の説明書は「実装メモとしては良い」が、「チームで開発できる仕様書」としては情報の層が足りません。改善版では、仕様の解像度を上げつつ、実装判断がぶれないように順番と粒度を整えるのが最優先です。
